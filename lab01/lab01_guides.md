# Лабораторна робота №1: Інструментарій сучасного веб-розробника

**Мета роботи**: Ознайомитися з базовим стеком інструментів для фронтенд-розробки, опанувати систему контролю версій Git та навчитися працювати з хмарними середовищами розробки.

## Етап 1: Налаштування локального середовища (VS Code)

1. **Інсталяція**: Завантажити та встановити **Visual Studio Code** або інший редактор коду (систему автоматизованої розробки): **WebStorm**, **Sublime Text**.

2. **Налаштування**: налаштуйте середовище для комфортної роботи з кодом та автоматизації рутинних операцій:

    - опція *Linked Editing*, дозволяє одночасно редагувати парний тег
    - опція *Files: Auto Save*, дозволяє не втратити внесені зміни у разі аварійного завершення роботи редактора (збереження відбувається автоматично після кожної зміни, або перемикання вікна)
    - прискорювач набору *Emmet*, дозволяє писати десятки рядків HTML за допомогою спеціальних шорткатів (коротких шаблонів). Він увімкнений за замовчуванням
    - опція *Format on Save* (Форматування при збереженні), вирівнює форматування команд, дужок та вставляє автоматично відступи у коді при збереженні змін
    - опція *Bracket Pair Colorization* (Кольорові дужки). Парні дужки фарбуються в один колір (наприклад, обидві жовті, або обидві сині). Це візуально розмежовує блоки коду.
    - опція Sticky Scroll, коли ви гортаєте код вниз, назва класу, функції чи тегу, в якому ви зараз перебуваєте, "прилипає" до верхньої частини екрана. Ви завжди розуміє контекст, який блок/тег/клас редагується в даний момент.

3. **Розширення**: Для комфортної роботи встановіть наступні плагіни:

    - *Live Server* — для миттєвого перегляду змін у браузері. Запускає локальний сервер для вашого проєкту одним кліком. Вам не потрібно щоразу оновлювати сторінку в браузері (F5) після редагування коду. Як тільки ви зберігаєте файл (Ctrl + S), браузер миттєво відображає зміни.
    - *ESLint* — для аналізу помилок у JS. Аналізує ваш JS-код на льоту та підкреслює потенційні помилки або порушення стандартів. Він підкаже, якщо ви забули оголосити змінну або використовуєте застарілий синтаксис, ще до того, як ви запустите скрипт у браузері.
    - *GitLens* — Git supercharged. Візуалізує історію змін прямо в коді. Ви бачите "inline blame" — біля кожного рядка написано, хто і коли зробив цей коміт. Також він додає зручний графічний інтерфейс для перегляду гілок та порівняння файлів, що значно полегшує розуміння роботи репозиторію.
    - *Error Lens*. Виводить повідомлення про помилки прямо в кінці рядка коду яскравим кольором. Вам не потрібно наводити мишкою на червоне підкреслення, щоб зрозуміти, що не так.

> Можна встановити інші плагіни, які полегшують роботу з середовищем. Перед встановленням плагіну переконайтесь, що він безпечний (відгуки, коментарі, рейтинг, кількість завантажень) та не дублює функціонал, який пропонується самим середовищем.

## Етап 2: Створення структури проєкту

Створіть каталог проєкту `lab01`, всередині каталогу додайте три файли: `index.html`, `style.css` та `script.js`. Відповідно до призначеного індивідуального варіанту скопіюйте вміст у відповідні файли проекту з репозиторію **ЛР1** на Github [kdg-auts / my-first-website / lab01](https://github.com/kdg-auts/my-first-website/tree/main/lab01)

`index.html` - код HTML: базова розмітка з заголовком, текстовим блоком та кнопкою.
`style.css` - код CSS: стилізація елементів (колір, шрифти, відступи).
`script.js` - код JavaScript: сценарій, який при натисканні на кнопку змінює текст, колір фону сторінки, розміри чи форму елементів на сторінці.

## Етап 3: Робота з DevTools (Інспектор)

Відкрийте файл `index.html` у браузері (якщо встановили плагін *Live Server*, запустіть його за допомогою кнопки **Go Life** на статусному рядку внизу праворуч).

Натисніть `F12` або `Ctrl+Shift+I` (або пункт `Inspect` в контекстному меню вмісту сторінки).

Дослідіть вкладки та панелі Developer Tools:

- *Elements*: Змініть колір тексту прямо в браузері.
- *Console*: Перевірте наявність помилок та виведіть повідомлення через console.log().
- *Sources*: Подивіться на завантажені файли стилів та скриптів.
- Styles: Подивіться на перелік правил стилістичного оформлення, що застосовуються до сторінки чи обраного елемента

## Етап 4: Git та GitHub

1. **Інсталяція**: завантажте та встановіть на вашому комп'ютері систему для контролю версій програмного забезпечення git [Git Web Page - Install](https://git-scm.com/install).

2. **Ініціалізація**: Виконайте команду `git init` у папці проєкту (в системній командній консолі, перейшовши до каталогу, або в консолі в середовищі проєктування, коли відкритий ваш проект ЛР1).

3. **Коміти**: Зробіть декілька послідовних комітів до репозитарію вашого проєкту. Зробіть перший коміт, додавши у *staging area* (`git add .`) новостворені файли вашого проєкту. Внесіть якісь зміни у окремі файли вашого проєкту, наприклад у `style.css`, змінивши колір фону якогось елементу чи колір тексту, та зробіть другий коміт. За бажанням можете зробити третій коміт після внесення додаткових змін у коді (наприклад, змінивши форматування).

4. **GitHub**: Створіть публічний репозиторій для вашого проєкту на сайті *Github*. За бажанням можете використати інший ресурс, наприклад *Gitlab* або *Bitbucket*.  Зв'яжіть локальну папку з віддаленою (команда `git remote add origin [URL to .git of repository]`, або `git remote set-url origin [new URL to .git of repository]`, якщо віддалений репозиторій вже був встановлений та треба змінити URL) та завантажте код (`git push`).

5. **Інструменти GitHub**: Ознайомтесь з можливостями та інструментами взаємодії з віддаленим репозиторієм. Створіть *Issue* з описом майбутньої фічі, створіть нову гілку, внесіть зміни та оформіть *Pull Request*. За бажанням, домовтесь з іншим студентом-напарником, щоб він вніс *Issue*/*Pull Request* до вашого проєкту, а ви - до його проєкту.

По кожному етапу мають бути наведені переконливі докази виконання завдання: знімки екрану середовища проєктування та результатів виводу у браузері (не робіть скріншот всього екрану, вікна мають бути зменшені до максимально компактного розміру, щоб текст на усіх зображеннях можна було однозначно прочитати без збільшення масштабу зображення), результати виводу в консолях, тощо.
4. **Розгорнуті висновки** по виконаній роботі із зазначенням складнощів, які виникали при виконанні завдань, та описом прийнятих рішень для їх усунення.
